# IBM Power (ppc64le) and IBM Z (s390x) Builds

## Overview

This document tracks the work needed to add hermetic Konflux builds for IBM Power (ppc64le) and IBM Z (s390x) architectures for the `services/uds_tokenizer` component. The P/Z builds must be part of the **same multi-arch image index** as the existing x86_64/arm64 builds.

## Current State

The `odh-llm-d-kv-cache-pull-request.yaml` builds for all 4 architectures:
- `linux/x86_64`
- `linux-m2xlarge/arm64`
- `linux/ppc64le`
- `linux/s390x`

x86_64/arm64 use `registry.access.redhat.com/ubi9/python-312:9.7` as the base image. ppc64le/s390x use `registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2` (which ships vllm + torch pre-installed).

The `Dockerfile.konflux` uses a BuildKit conditional `FROM` pattern based on `TARGETARCH` to select the appropriate base image per architecture.

The `uv.lock` resolves for all 4 architectures with dedicated resolution forks:
```
resolution-markers = [
    "... platform_machine == 'x86_64' and sys_platform == 'linux'",
    "... platform_machine == 'aarch64' and sys_platform == 'linux'",
    "... platform_machine == 'ppc64le' and sys_platform == 'linux'",
    "... platform_machine == 's390x' and sys_platform == 'linux'",
]
```

A single `requirements.txt` (generated by `uv export`) handles all 4 architectures via platform markers. On ppc64le/s390x, vllm/torch entries are skipped (no markers for those arches) since they come from the vllm-spyre base image.

The `requirements.txt` includes `--index-url` pointing to the internal rhoai PEP 503 index, so hermeto/cachi2 downloads packages from the internal index rather than PyPI. This avoids PyPI's 2GB+ CUDA-enabled torch wheels. All 138 packages are available on the rhoai index.

## Key Decision: vllm on ppc64le/s390x

vllm has **no pre-built wheels for ppc64le/s390x** — neither upstream nor on the internal py-index. The solution is to use the **vllm-spyre base image** for ppc64le/s390x builds, which ships vllm pre-installed.

On x86_64/aarch64, vllm is installed via pip from the internal rhoai-wheels index (`vllm==0.16.0+rhaiv.3`). torch, torchaudio, and torchvision are pinned as direct dependencies and routed to the rhoai index via `[tool.uv.sources]` to ensure the CPU-only builds are used (not PyPI's CUDA-enabled wheels).

## vllm-spyre Base Image

The image `registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2` is a multi-arch image (ppc64le, s390x, amd64) that ships with:
- Python 3.12, virtualenv at `/opt/app-root` (same layout as UBI python images)
- vllm 0.16.0+rhaiv.8 (Spyre variant)
- torch 2.7.1
- 218 PyPI packages total (from SBOM inspection)

Python paths are **identical** to UBI python images:
- Site-packages: `/opt/app-root/lib/python3.12/site-packages`

- `VIRTUAL_ENV=/opt/app-root`

## Implementation (Completed)

### Requirements Strategy: Single `requirements.txt` for All Architectures

**No separate `requirements-pz.txt` is needed.** By extending the `uv.lock` resolution to include ppc64le/s390x, a single `requirements.txt` with platform markers handles all 4 architectures.

#### How it works

The `pyproject.toml` has vllm, torch, torchaudio, and torchvision as platform-conditional dependencies (x86_64/aarch64 only):
```toml
"vllm==0.16.0+rhaiv.3 ; platform_system == 'Linux' and platform_machine == 'x86_64'",
"vllm==0.16.0+rhaiv.3 ; platform_system == 'Linux' and platform_machine == 'aarch64'",
"torch==2.10.0 ; platform_system == 'Linux' and (platform_machine == 'x86_64' or platform_machine == 'aarch64')",
"torchaudio==2.10.0 ; ...",
"torchvision==0.25.0 ; ...",
# No entry for ppc64le/s390x — these come from the vllm-spyre base image
```

These packages are routed to the internal rhoai PEP 503 index via `[tool.uv.sources]`:
```toml
[[tool.uv.index]]
name = "rhoai"
url = "https://packages.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/"
default = true

[[tool.uv.index]]
name = "pypi"
url = "https://pypi.org/simple/"

[tool.uv.sources]
vllm = { index = "rhoai" }
torch = { index = "rhoai" }
torchaudio = { index = "rhoai" }
torchvision = { index = "rhoai" }
```

The rhoai index is set as `default = true` (first-match strategy), so all packages are sourced from the internal index when available, with PyPI as a fallback. The `[tool.uv.sources]` explicitly routes vllm/torch/torchaudio/torchvision to the rhoai index to prevent uv from picking PyPI's CUDA-enabled torch wheels (which list nvidia-* dependencies).

When uv resolves for ppc64le/s390x, it:
- **Skips vllm** (no dependency entry for those platforms)
- **Skips torch/torchaudio/torchvision** (transitive deps of vllm, which is absent on ppc64le/s390x)
- **Skips vllm-exclusive transitive deps** (blake3, llguidance, etc.)
- **Includes all other deps** (grpcio-tools, shortuuid, etc.) with ppc64le/s390x markers

At build time on ppc64le/s390x:
- pip reads `requirements.txt`, skips entries whose markers don't match
- Only installs packages not already in the vllm-spyre base image
- Result: base image provides vllm+torch+deps, pip provides the rest

#### intel-cmplr-lib-ur exclusion

`intel-cmplr-lib-ur` is a transitive dependency of `intel-openmp` (which torch needs on x86_64 for `libiomp5.so`). It provides Intel Unified Runtime adapters (Level Zero, OpenCL) for Intel GPU offloading — **not needed for CPU-only torch**. It fails cachi2 hermetic prefetch, so it is excluded via `override-dependencies`:

```toml
override-dependencies = [
    "intel-cmplr-lib-ur ; python_version == '0'",
]
```

**Note:** `uv export` does not respect this override, so after export you must manually remove the line:
```bash
sed -i '/^intel-cmplr-lib-ur/d' requirements.txt
```

#### Steps to regenerate

```bash
cd services/uds_tokenizer/
uv lock --python 3.12
uv export --format requirements-txt --output-file requirements.txt --python 3.12 --no-dev
sed -i '/^intel-cmplr-lib-ur/d' requirements.txt
python3 generate_hashes.py
```

**Notes:**
- `--python 3.12` is required because the internal index only has cp312 wheels.
- `uv export` includes hashes for most packages from the PEP 503 index, but a few packages that fall back to PyPI may have missing hashes. The `generate_hashes.py` script fetches hashes from the PEP 503 API for any remaining packages without hashes, and prepends `--index-url` to `requirements.txt` so hermeto/cachi2 downloads from the internal index.
- `uv export` ignores `override-dependencies`, so `intel-cmplr-lib-ur` must be removed manually via `sed`.

### Dockerfile: Unified with Conditional Base Image

Since P/Z builds must be part of the **same image index**, `Dockerfile.konflux` uses BuildKit's conditional `FROM` pattern based on `TARGETARCH`:

- **x86_64/arm64**: Use `registry.access.redhat.com/ubi9/python-312:9.7`
- **ppc64le/s390x**: Use `registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2`

Both images share the same Python virtualenv layout (`/opt/app-root/`), making multi-stage `COPY` compatible across all architectures.

```dockerfile
ARG TARGETARCH

# --- Builder bases per architecture ---
FROM registry.access.redhat.com/ubi9/python-312:9.7 AS builder-amd64
FROM registry.access.redhat.com/ubi9/python-312:9.7 AS builder-arm64
FROM registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2 AS builder-ppc64le
FROM registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2 AS builder-s390x

FROM builder-${TARGETARCH} AS builder
# ... single requirements.txt — platform markers handle arch differences

# --- Runtime bases per architecture ---
FROM registry.access.redhat.com/ubi9/python-312:9.7 AS runtime-amd64
FROM registry.access.redhat.com/ubi9/python-312:9.7 AS runtime-arm64
FROM registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2 AS runtime-ppc64le
FROM registry.redhat.io/rhaii-early-access/vllm-spyre-rhel9:3.4.0-ea.2 AS runtime-s390x

FROM runtime-${TARGETARCH} AS runtime
# ... rest of Dockerfile unchanged — same virtualenv paths work for both bases
```

```yaml
- name: build-platforms

  - linux/x86_64
  - linux-m2xlarge/arm64
  - linux/ppc64le

- name: prefetch-input
```

Since we use a single `requirements.txt`, the prefetch configuration is simple — the new arches are added to the `binary.arch` field.

## Risks

1. **vllm version difference on ppc64le/s390x**: x86_64/aarch64 install `vllm==0.16.0+rhaiv.3` via pip, while the vllm-spyre base image ships `0.16.0+rhaiv.8`. Minor variant difference — should be compatible but worth verifying.
2. **torch version mismatch on ppc64le/s390x**: x86_64/aarch64 install `torch==2.10.0`, while vllm-spyre ships `torch==2.7.1`. Verify torch API compatibility if the service uses torch directly.
3. **OS base mismatch**: x86_64/arm64 use UBI 9, ppc64le/s390x use RHEL 9 (vllm-spyre). Both are RHEL 9-based so system library differences should be minimal.
4. **Hash coverage**: Most packages get hashes natively from `uv export` via the PEP 503 index. A few packages that resolve from PyPI as fallback may lack hashes; `generate_hashes.py` fills these in from the PEP 503 API.
5. **`uv export` limitations**: `uv export` does not respect `override-dependencies`. The `intel-cmplr-lib-ur` override requires a post-export `sed` fixup (see regeneration steps above).
6. **Index routing**: `[tool.uv.sources]` only applies to direct dependencies. torch/torchaudio/torchvision are pinned as direct deps specifically to enable index routing. Without this, uv's `first-match` strategy may resolve them from PyPI (CUDA-enabled, with nvidia-* deps).
